* DONE Effects as thunks of logic
CLOSED: [2026-03-12 Thu 17:47]
stop parsing effects in line when applying them. when effects (spell, ability) are on the stack (or probably being carried out instantly) they should be (generally) anonymous functions composed of well tested reusable components like destroy(target), draw(cards), etc. This way we can produce arbitrarily complex cards and especially triggered and continuous effect abilities without implementing a new "effects" language to parse.

This entails difficulties in selecting targets for effects that need targets when putting them on the stack.
* DONE TargetSpec
CLOSED: [2026-03-12 Thu 17:47]
TargetSpec solves these difficulties. We create a boolean algebra of target legality.

So a targetted ability has a well defined effect (from the above work) and if it needs a target, like "destroy target non black creature" we need to be able to specify "object satisfies: zone = in play, type = creature, color-identity = non-black"

So to produce a trigger in general, we must describe the effect as a hunk of code and the target predicate as a composed predicate. Then the trigger is ready for the trigger owner to decide targets (same for spell is ready for spell caster to select targets to cast). This needs to be aware of the trigger source object (same for spells) to allow strategic choice.

All targettable objects in the game should be "GameObjects" or "SimObject" whatever. These should have totally generic properties shared between players and cards (could be protection from X for example, not yet defined). But more interestingly, targettable things in general are:
** objects
*** players
*** spells on stack
e.g. counter with fow / daze
*** abilities on stack
e.g. counter with stifle / consign to memory
*** creatures in play
e.g. destroy with fatal push / snuff out
*** permanents in play
e.g. bounce with petty theft
*** lands in play
e.g. destroy with wasteland.

** predicates / properties
*** object type
*** color
*** mana value
*** color
*** attacking/blocking
*** tapped/untapped
*** more ...

* DONE Object ID
CLOSED: [2026-03-12 Thu 17:47]
Currently we only identify objects in play by their name.
This breaks under duplicates.

Instead we must give each object (player, card, object on stack, token, emblem, etc.) a unique ID. Ideally we store all game objects in a handful of flat associative data structures keyed by ID and associate them with the object itself and perhaps some super high level identifying information like current zone. Then we can have a "Cards" map ID -> Card,Owner,Zone or something and implement "change zones" as changing the zone and firing a zone change event. (Zone being library, hand, stack, battlefield, graveyard, exile). Zone can also stay or be a property of the Card. Abilities on stack would be relatively simpler, not having a Zone besides stack, but still eneding to be indexed by ID.
